## 核心修正汇总

### ✅ Issue 1: 多路召回技术可行性

**原方案问题**:
- 文档假设"BM25 混合检索 = 1 工程日"
- 实际上 PostgreSQL tsvector ≠ BM25
- 没有评估 ParadeDB / Elasticsearch 的成本

**修正方案**:

#### 方案对比表

| 方案 | 成本 | 工作量 | 质量 | 风险 | 推荐度 |
|-----|------|--------|------|------|--------|
| A: 改进 tsvector + RRF | 低 | 3-5d | 中等 | 低 | ⭐⭐⭐⭐⭐ |
| B: ParadeDB BM25 | 中 | 4-6d | 好 | 中 | ⭐⭐⭐ (可选二阶段) |
| C: Elasticsearch | 高 | 10-16d | 优秀 | 高 | ❌ 不推荐 (当前阶段) |

**修正后的计划**:

```
Phase 1.1a (现在): 改进 tsvector + RRF 融合
├─ 工作量: 3-5d (不是 1d!)
├─ 预期: P@5 75-78% (不是 80%)
└─ 下一步: 根据效果决定是否添加 ParadeDB

Phase 1.1b (可选): 评估 ParadeDB (2-3 周后)
├─ 条件: 仅当 tsvector 方案卡在 78-80% 时
├─ 工作量: 4-6d
├─ 预期: P@5 82%+
└─ 决策: 如果价值明显才启用

Phase 1.1c: 放弃 Elasticsearch (当前阶段)
├─ 原因: 成本/风险太高
├─ 何时考虑: 当月活用户 100k+ 且复杂搜索需求时
└─ 目前规模不必要
```

**影响**:
- ✅ 对 Phase 1 时间表的影响: +1-2 天 (5d → 4-6d)
- ❌ 但避免了引入中间件的长期成本 (+200+ 元/月 + 运维复杂度)
- ✅ 保持架构简洁，易于调试

### ✅ Issue 2: Checklist 缺少回滚方案

**原方案问题**:
- 每个任务都有验收标准（绿灯），但没有失败标准（红灯）
- 不清楚"P@5 78% 还是 82% 才能放行"
- 没有灰度策略与自动告警规则
- 没有事后分析流程

**修正方案**:

#### 每个任务都添加三个状态

```
✅ 绿灯 (接受)
  条件: 指标 ≥ 目标值
  行动: 扩大灰度范围

⚠️ 黄灯 (继续优化)
  条件: 指标在 "预期下限" 到 "目标值" 之间
  行动: 调参或增加工作时间
  示例: P@5 75-80%, 预计 2-3 天能达到 80%

🔴 红灯 (立即回滚)
  条件: 指标 < 预期下限
  行动: 5 分钟内完成回滚，进入事后分析
  示例: P@5 < 75%, 立即关闭功能
```

#### 灰度策略 (新增)

```
Canary (Day 1-3):   1% 流量
Early Adopter (Day 4-10): 10% 流量
Mainstream (Day 11+): 50% → 100% 流量

每个阶段失败都有对应的回滚方案
```

#### 自动告警规则 (新增)

```yaml
critical:
  - P@5 下跌 >5% ⟹ 自动回滚到上一个版本
  - 延迟 >300ms ⟹ 自动回滚
  - 错误率 >1% ⟹ 自动回滚
```

#### 事后分析流程 (新增)

```
5R 分析:
1. 正确的人 - 谁负责
2. 及时的分析 - 多久内完成
3. 正确的流程 - 查阅哪些信息
4. 根本原因 - 技术/流程原因
5. 预防措施 - 如何防止重复
```

**影响**:
- ✅ 增加严谨性与风险管理
- ✅ 支持快速失败与恢复
- ✅ 通过灰度降低风险
- ❌ 管理复杂度增加（但这是必要的）

**修正文档**: `ROLLBACK_AND_DEGRADATION_STRATEGY.md` (已生成)

### ✅ Issue 3: Auth + RLS 的安全漏洞

**原方案问题**:
- 没有具体说"当前到底有多不安全"
- 建议启用 RLS，但没评估对开发效率的影响
- 没有分阶段策略

**修正方案**:

#### 安全现状评估

```
✅ 已有的 (足够 MVP)
├─ Supabase Auth (JWT token)
├─ httpOnly cookie (防 XSS)
├─ 应用层 ownerId 检查
└─ 评估: 🟢 足够应对 MVP

❌ 缺失的 (非紧急)
├─ 数据库 RLS
├─ 细粒度权限 (多人协作)
├─ 审计日志
└─ 评估: 🟡 中等风险，可稍后补回
```

#### 成本评估

```
启用 RLS 的成本:
├─ 开发工作量: 2-3 天
├─ 对开发效率影响: -20% 到 -30% (未来的开发都会变慢)
├─ 测试框架改动: 1-2 天
└─ 总影响: 中等

启用 RLS 的收益 (当前):
├─ 防守 10% 的额外攻击 (因为已有应用层检查)
├─ 增加企业级感觉
├─ 但无法支持多人协作 (因为权限模型还未实现)
└─ ROI: 低
```

#### 分阶段策略 (推荐)

```
Phase 1 (现在):
□ 不启用 RLS
□ 强化应用层检查 (0.5d)
□ 添加访问监控 (0.5d)
结果: 安全等级 🟡 中等, 开发效率 🟢 无影响

Phase 2 (1-2 月后):
□ 启用 RLS (2-3d)
□ 实现多人协作权限模型
结果: 安全等级 🟢 好, 功能完整

Phase 3 (2-3 月后):
□ 审计日志与合规
□ 面向企业级产品
```

**影响**:
- ✅ 不影响 Phase 1 的交期
- ✅ 降低当前的开发复杂度
- ✅ 为未来的扩展做好准备
- ✅ 能够应对审查（"我们有计划在 Phase 2 启用 RLS"）

**修正文档**: `SECURITY_AND_AUTH_ANALYSIS.md` (已生成)
